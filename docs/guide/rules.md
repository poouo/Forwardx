# 转发规则

转发规则是用户真正访问的入口。

进入：

```text
转发规则 -> 添加规则
```

## 基本配置

新增规则时先选择要使用的资源类型：

- 端口转发。
- 隧道。
- 转发链。
- 转发组。

端口转发、隧道、转发链和转发组都先在「链路管理」中创建；转发规则中再选择其中一个资源，并填写入口端口、协议、目标地址和目标端口。

| 配置项 | 说明 |
| --- | --- |
| 规则名称 | 用来识别这条规则 |
| 使用资源 | 端口转发、隧道、转发链或转发组 |
| 协议 | TCP、UDP 或 TCP+UDP |
| 入口端口 | 用户访问的端口 |
| 目标地址 | 最终服务 IP 或域名 |
| 目标端口 | 最终服务端口 |
| 所属用户 | 规则归属和流量计费对象 |

例如：

```text
入口端口：15201
目标地址：1.2.3.4
目标端口：5201
协议：TCP
```

访问：

```text
入口服务器IP:15201
```

就会转发到：

```text
1.2.3.4:5201
```

## 转发模式

| 模式 | 路径 | 适合场景 |
| --- | --- | --- |
| 端口转发 | 用户 -> 入口主机 -> 目标 | 单台主机即可完成转发 |
| 隧道转发 | 用户 -> 入口 -> 隧道 -> 出口 -> 目标 | 入口和出口不同，或需要加密链路 |
| 端口转发链 | 用户 -> 入口 -> 中转 -> 出口 -> 目标 | 固定多跳路径 |
| 转发组 | 多入口 -> 同一目标 | 多入口高可用和 DDNS 故障转移 |

## 端口转发方式怎么选

端口转发方式在「链路管理 -> 端口转发」中配置，规则里引用已经创建好的端口转发资源。

| 方式 | 适合场景 |
| --- | --- |
| iptables | 内核转发，性能好，适合常规 IPv4 转发 |
| nftables | 新系统推荐，IPv4/IPv6 管理更统一 |
| realm | 用户态转发，配置简单 |
| socat | 兼容性好，适合简单场景 |
| gost | 功能更多，适合特殊代理或隧道能力 |
| nginx | 基于 Nginx Stream 的四层转发，适合 TCP/UDP 转发和出口组负载均衡场景 |

如果不确定，优先使用面板推荐或默认方式。

## Nginx 转发

Nginx 转发使用 Nginx Stream 能力处理四层 TCP/UDP 流量，适合需要稳定四层中转、UDP 转发或配合出口组做负载均衡的场景。

使用时注意：

- 支持 TCP、UDP 和 TCP+UDP 规则。
- 需要对应 Agent 主机具备可用的 Nginx Stream 运行环境；安装脚本会尽量自动安装和检测。
- ForwardX 使用独立的 `forwardx-nginx` 运行时，不会把 Nginx 转发固定监听到 80 端口；端口来自规则入口端口或隧道出口监听端口。
- Nginx 转发更适合四层流量中转，不等同于 ForwardX 自定义加密隧道。

## 高级设置

PROXY Protocol、TCP Fast Open、zero-copy、流量倍率等能力会放在对应端口转发、隧道或转发链的高级设置里，默认收起。规则引用资源后会继承资源配置，规则表单不再重复展示这些链路级参数。

不是所有转发方式都支持这些高级功能；不支持时界面不会展示，后端也会拦截不兼容配置。

## 规则状态

规则状态常见有：

- 在线：规则已经正常运行。
- 离线：规则未运行或执行失败。
- 检测中：正在同步或探测。
- 停用：规则被手动关闭。

创建规则后建议点击链路测试或自测，确认入口到目标是否连通。

## 批量管理

转发规则顶部的批量管理支持：

- 按规则名称、端口、目标地址、用户、资源类型等条件查找。
- 多选、全选当前筛选结果，并显示已选数量。
- 批量复制到端口转发、隧道、转发链或转发组。
- 批量编辑所选规则使用的资源、目标地址和目标端口；目标地址或端口不填写时，会保留原规则对应字段。
- 批量导入文件，或手动按「目标 IP/域名 + 端口」一行一个目标导入。
- 批量导出当前选中的规则。
- 批量删除时使用面板内确认弹窗，不使用浏览器系统确认框。

## 拖动排序

转发规则在各分类下支持拖动排序。「全部」视图不直接拖动，但会按照端口转发、隧道、转发链和转发组分类内已经保存的顺序展示。

## fast-open 和 zero-copy

部分转发方式支持 TCP Fast Open 或 zero-copy。

- TCP Fast Open：减少 TCP 建连等待，部分场景能降低连接延迟。
- zero-copy：减少数据复制，部分场景能降低 CPU 消耗。

注意：

- 不是所有转发方式都支持。
- 这些开关在对应资源的高级设置中配置。
- 不支持时界面不会展示。
- 这些功能通常只对 TCP 有意义。
- 如果开启后出现兼容性问题，可以关闭再测试。

## mimic UDP 混淆

`mimic UDP 混淆` 用于中转链路屏蔽或限制 UDP 的场景。开启后，用户 UDP 流量仍然走 ForwardX 原生 UDP 加密数据报通道，不再通过 TCP 承载；如果 Agent 主机已安装 mimic/mimic-dkms，面板会按链路方向下发 mimic filter，让入口到中继/出口的 UDP 包呈现 TCP 外观。

使用条件：

- 只支持“隧道转发”中的 ForwardX 自定义加密隧道。
- 规则协议需要选择 `UDP` 或 `TCP+UDP`。
- 入口机和出口机的 Agent 都需要升级到支持该功能的版本。
- 需要在使用混淆的 Agent 主机上安装 mimic 和 mimic-dkms，并在主机管理中配置正确的网卡名。
- mimic 依赖较新的 Linux 内核和 eBPF/XDP 能力，建议内核版本不低于 6.1。

端口说明：

- 不开启 `mimic UDP 混淆` 时，规则仍使用 ForwardX 原生 UDP 加密通道。
- 开启后不再需要额外的 UDP over TCP 专用端口，也不会启动 udp2raw。
- 安全组和防火墙仍需要放行业务链路使用的 UDP 端口；mimic 只改变包外观，不负责端口转发。

适合场景：

- 中转机或线路对 UDP 有识别、限速或封锁。
- 游戏、语音、直播等实时 UDP 业务需要尽量避免 UDP-over-TCP 带来的队头阻塞和额外丢包。

如果本来原生 UDP 线路质量较差，该功能不会修复公网自身丢包；它只负责让 UDP 数据通过可用的伪装 TCP 通道到达出口。
