# Changelog

## [2.3.213] - 2026-07-03

### 修复

- 修复 Agent 注册或重连时面板整批清空运行态，导致规则较多时长期显示黄色、随后又重复刷新下发的问题；现在仅失效期望状态缓存，保留可验证的本地运行态。
- 修复面板无法获取 Agent 实际 CPU 占用、数值长期显示为 `0` 的问题；Agent 改为通过 `/proc/stat` tick 差值计算真实 CPU 百分比，并仅在不可用时回退到 loadavg 估算。
- 修复面板日志切换查看时分类数量会短暂变为 `0` 再恢复的问题，日志分类统计现在会保留上一份有效 summary。
- 修复首页流量饼图、规则活跃率和规则负载进度条颜色会被主题色覆盖的问题，相关统计图恢复固定业务色展示。

### 优化

- Agent 期望状态下发新增签名缓存和重发节流，配置未变化时不再在短时间内反复向 Agent 下发大包，仅在配置变化、活跃同步或周期校验时重发。
- Agent 支持采纳本地已存在且与期望状态匹配的规则/隧道运行态，并主动回报 running，减少 200 条到 500 条规则场景下的重复 apply。
- Agent 期望状态失败后增加 5 分钟退避，避免失败规则在每轮心跳中高频重试造成 CPU 和 shell 压力。
- Agent 流量采集间隔根据规则数量和采集耗时自适应退避，规则较多或采集较慢时自动拉长间隔，降低高规则数场景的持续负载。
- Agent ping 探测优先使用 Go 原生 IPv4 ICMP 实现，权限或 IPv6 不支持时才回退系统 `ping`，减少对 shell 指令的依赖；tcping 继续使用 Go 原生并发拨测。

### 版本

- 面板版本升级至 `2.3.213`。
- Agent 目标版本升级至 `2.2.135`。
- Android APP 版本保持 `2.3.77`。

## [2.3.212] - 2026-07-03

### 修复

- 修复大量规则或协议守护规则反复同步时，Agent 删除不存在的 systemd 托管服务仍会持续触发 `daemon-reload`，导致 PID 1 / systemd CPU 占用升高的问题。
- 修复协议守护 backend apply 时会先删除当前要启动的 `forwardx-*-guard-*` 服务再重建，造成守护服务反复 stop/start、规则状态抖动和心跳延迟的问题。
- 修复已删除的 `forwardx-socat-guard-*` 等失败单元残留在 systemd failed 状态后，后续清理没有执行 `reset-failed`，可能持续干扰状态判断的问题。

### 优化

- 新 Agent 心跳协议新增 desired-state 期望状态同步：面板只声明当前主机应该存在的规则/隧道/runtime 状态，Agent 本地按签名 diff 后只执行新增、删除或配置变化的项，避免 190+ / 500+ 规则场景下每轮心跳重复下发并执行大批 shell。
- Agent 托管服务写入改为内容比对：unit 文件无变化时不再执行 `systemctl daemon-reload`。
- Agent 为托管服务记录动作签名，配置未变化且服务仍处于 active 状态时跳过重复 restart，降低 190+ / 500+ 规则场景下的 systemctl 与 shell 阻塞。
- Agent runtime 同签名同步从 5 分钟重跑一次调整为 30 分钟兜底，并在本地配置对应服务仍 active 时直接跳过，减少大量规则场景下的周期性 shell 压力。
- 面板下发的托管服务写入/删除命令改为幂等执行，重复 apply/remove 会尽量退化为低成本 no-op。

### 版本

- 面板版本升级至 `2.3.212`。
- Agent 目标版本升级至 `2.2.134`。
- Android APP 版本保持 `2.3.77`。

## [2.3.211] - 2026-07-03

### 修复

- 修复大量规则同步时 Agent action 队列容量过小，规则数接近或超过队列上限后心跳线程可能被入队阻塞，导致面板出现频繁离线/上线抖动的问题。
- 修复大量 gost 规则批量恢复时，普通规则 action 会重复触发 runtime reload，导致 190+ / 500+ 规则场景恢复耗时过长的问题；gost runtime 统一由 runtime-sync 重载，单条规则仅处理端口清理、计数链和状态回报。
- 修正 Docs workflow 中 GitHub Pages deploy timeout 超过 `actions/deploy-pages` 允许最大值导致的误导性 warning。

### 优化

- Agent 在 action 队列繁忙时改为每 10 秒发送 `busy` 保活心跳，面板只更新在线状态和主机指标，不重复下发运行时任务。
- Agent action 队列扩容到 4096，并在极端满队列时后台入队，避免高压规则同步阻塞主心跳循环。
- Agent 将同一个 action 内的多条 shell 命令合并为一次批量执行，减少大量规则场景下反复启动 shell 的开销。

### 版本

- 面板版本升级至 `2.3.211`。
- Agent 目标版本升级至 `2.2.133`。
- Android APP 版本保持 `2.3.77`。

## [2.3.210] - 2026-07-02

### 新增

- 系统设置新增“允许多设备在线”开关，默认关闭；开启后同一账户可在多台设备同时访问后台，不再限制为单一活跃会话。
- 个性化配置新增“默认配色”预设，内置墨色、松石、森绿、紫藤、暖阳、樱粉六套主题，选择后可通过“保存配色”统一保存并同步到按钮、选中态、提示框和侧边栏主色。

### 修复

- 修正规则/隧道批量重配或大量 action 排队时，Agent 主循环会一直延后普通心跳，导致面板 90 秒在线 TTL 判定为离线、随后又因下一次心跳恢复为在线的假离线/上线抖动问题。
- Agent 在 action 队列持续繁忙时会发送 `busy` 保活心跳，面板收到后只更新在线状态和主机指标，不会重复计算或下发运行时任务，避免忙时保活反过来放大队列。
- 主机在线判定 TTL 从 90 秒放宽到 150 秒，并同步主机列表、Token 列表和首页统计口径，降低单次慢请求或短暂网络抖动造成的误判。
- 修正同一账户在其他设备登录后，当前设备 Cookie 会被立即判定失效的问题；关闭“允许多设备在线”时改为最近活跃会话租约，仅在不同设备同时访问后台时触发单设备限制。
- 修复大量 gost / runtime 规则批量同步时，Agent 将过长 shell 命令直接传给 `sh -lc` 导致 `argument list too long`、规则持续 `running=false` 且心跳被 action 队列拖慢的问题；超长命令会自动写入临时脚本再执行。

### 优化

- 系统设置中的“主机监控配置”改为左侧独立卡片，右侧新增“登录会话配置”卡片并单独保存，减少单个配置区域横向占用。
- 系统设置、主机管理、链路管理、支付对接、余额/营销、用户管理、套餐管理和转发规则分类标签切换改为统一的胶囊式滑动选中效果，切换时选中态平滑滑动，替代原来的硬切换视觉。
- 清理面板中固定 sky/indigo/violet 等蓝紫色调的品牌类样式，改为跟随主题主色；个人资料页 Telegram/2FA 未配置提示和退出登录按钮也会随主题变化。
- 默认配色卡片选中图标固定占位，避免选中后文字重新排列；主题保存成功后立即写入前端 CSS 变量，减少按钮闪烁并确保颜色实际生效。
- 胶囊式滑动标签组件的选中背景和选中文字改为跟随主题主色，系统设置、主机管理、链路管理等所有复用位置都会同步变化。
- 仪表盘、余额与营销、用户管理和流量计费配置的统计卡图标恢复彩色渐变背景，避免墨色主题下统计入口变成单一深色。
- 左侧侧边栏恢复原有圆角块选中风格，去除不自然的选中滑动动画。
- 网络测试提示、主机管理统计卡、公共主机监控统计卡和主机分组筛选恢复主题色阶装饰，避免直接去色后变成灰色块。
- 默认配色新增“樱粉”泡泡糖粉主题，并让背景轻微渐变使用主题 chart 色阶，切换主题时页面背景会随当前主题协调变化。
- 公开首页示例模板会按当前默认配色生成，登录页和默认公开首页也会同步读取个性化主题色。

## [2.3.209] - 2026-07-02

### 新增

- 隧道和端口转发链新增“流量倍率”，默认 `1`，支持 `0.01` - `50`；普通转发规则按倍率计入用户流量配额，流量按量计费仍使用独立计费设置，不受该倍率影响。
- 转发规则选择隧道或端口转发链时新增倍率标识，并通过不同颜色区分倍率状态，便于创建规则时直接确认该入口的配额倍率。
- 新增用户累计流量和转发规则累计流量计数表，普通流量统计可查看长期累计值，不再依赖仅保留近 3 天的明细数据。

### 修复

- 修正 Docker 部署升级时自定义宿主机映射端口可能被重新写回 `9810` 的问题；Docker 升级脚本会优先读取当前容器真实端口映射，其次读取旧 `docker-compose.yml` 和 `.env`。
- 修正 Docker 容器环境可能被误判为本地部署、导致后台允许修改容器内 Web 监听端口的问题；Docker 下容器内固定监听 `3000`，宿主机访问端口由端口映射管理。
- 修正部分流量汇总、排行和 Telegram 查询仍按近 3 天口径展示的问题，相关总量改为读取累计计数。

### 优化

- 转发规则中的隧道下拉展示精简为用户备注、端口区间和流量倍率，去除多余拓扑文字，降低选择时的信息噪音。
- PROXY Protocol 配置改为按需开启，默认关闭且不展示具体配置项；关闭时会清空相关接收 / 发送选项，让规则编辑界面更整洁。
- 系统设置的 Web 服务监听端口卡片在 Docker 部署下改为只读展示，按钮不可点击，并明确提示通过部署目录 `.env` 的 `PORT` 或 `docker-compose.yml` 端口映射调整。
- 主机、链路测试和地图提示中所有国旗展示去除外层边框和阴影，直接展示国旗图片，整体视觉更轻。
- 数据库迁移、备份导入导出和启动迁移流程纳入累计流量计数表，保留长期累计值，同时仍按保留策略清理可再生成的明细数据。
